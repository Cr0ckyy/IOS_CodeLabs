Computer program http://en.wikipedia.org/wiki/Computer_program

A computer program is a sequence of instructions written to perform a specified task. The program has an executable form that the computer can use directly to execute the instructions. The same program in its human-readable source code form, from which executable programs are derived, enables a programmer to study and develop its algorithms.

Debugging http://en.wikipedia.org/wiki/Debugging

Debugging is a methodical process of finding and reducing the number of bugs, or defects, in a computer program, thus making it behave as expected.

Logging http://en.wikipedia.org/wiki/Logfile

Logging is the act of recording events that occur when a piece of software executes. This can be used as signposts for program execution or to produce a snapshot of variable values.

Breakpoints http://en.wikipedia.org/wiki/Breakpoint

A breakpoint is an intentional stopping or pausing place in a program, put in place for debugging purposes.

Stepping http://en.wikipedia.org/wiki/Program_animation

Stepping refers to the process of executing code one "line" at a time. The programmer may examine the state of the program, machine, and related data before and after execution of a particular line of code. This allows evaluation of the effects of that statement or instruction in isolation and thereby gain insight into the behavior (or misbehavior) of the executing program.

Syntax error / compile-time error http://en.wikipedia.org/wiki/Syntax_error

In computer science, a syntax error is a mistake in the use of the syntax of a particular programming language. Syntax errors are typically caught by the compiler, and prevents the creation of the executable program.

Logical error / run-time error http://en.wikipedia.org/wiki/Logic_error

In computer programming, a logic error is a bug in a program that causes it to operate incorrectly. A logic error produces unintended or undesired output or other behaviour. Logical errors cannot be caught by the compiler, and are usually only discovered when the program is executed.

In iOS development, print statements are used for logging to the debug area. Note that print statements should only be used when debugging.

Logging as an indicator

Use print statements as an indicator that a particular method has been called, or that one or more statements have been executed. The following print statement prints out the file name, method name and source code line number to the debug area when it is called.

print(#file, #function, #line)

For example, given the following piece of code:

import UIKit
class ViewController: UIViewController {
  override func viewDidLoad() {
    super.viewDidLoad()
    print(#file, #function, #line)
  }
}

If we see the following in the debug area:

/Users/solar/Desktop/SampleProject/SampleProject/ViewController.swift viewDidLoad() 5

We can say that the following is true:

  • the viewDidLoad method in the ViewController class was called
  • statements belonging to the viewDidLoad method before and including line 5 were executed (i.e. lines 4 and 5 were executed)

Logging to print out values of variables

Use print statements to print out the value of one or more variables.

For example, given the following piece of code:

import UIKit
class ViewController: UIViewController {

  override func viewDidLoad() {
    super.viewDidLoad()
    print(#file, #function, #line)

  @IBAction func actButton(_ sender: UIButton) {
    print("The text in the text field is \(outTextField.text!)")
  }
}

If we see the following in the debug area after we press the button:

The text in the text field is john

We can say that the following is true:

  • the actButton IBAction method is called when the button is pressed, i.e. the button is connected to the actButton IBAction method
  • the text in the text field is john

The debugging process can be broken down into 5 parts:

  1. Identify the steps to reproduce the error
  2. Reproduce the error
  3. Find / identify the code involved
  4. Fix the code
  5. Test that the error no longer exists

Before a bug can be fixed, the developer must first be able to reproduce the bug. Otherwise,

  • It may be difficult for the developer to find / identify the code involved
  • After fixing the code, it may be difficult to test that the error no longer exists

Hence, the first thing to do in the debugging process is to identify the steps to reproduce the bug. It is important to reproduce the bug using the least possible number of steps because that will help to zero in on where the error is.

Once the developer knows the steps to reproduce the bug, the next thing to do is to reproduce the bug, and find the code which is causing the error. One of the techniques to find the faulty code is to read the error messages in the IDE. Other techniques include logging, and breakpoints and stepping.

After understanding the cause of the error, the code needs to be fixed to address the error. When that is done, the code needs to be tested again to ensure that the bug no longer exists.

Reading error messages

Error messages in the IDE or the debug area may offer a hint about which line of code was being executed when the error occurred. Some error messages are easy to understand while some are not. The more errors you encounter and learn from, the easier it becomes to understand and fix the errors. At its best, understanding all the possible errors will help you avoid writing code which causes those errors in the first place.

When it comes to reading error messages in Xcode, you need to look at the error messages in the editor window and also in the debug area.

Example 1: examine the following scenario where an application crashes:

The error indication on line 7 of the editor window says Fatal error: Index out of range but the information in the debug area seems to be referring to some Swift library files. In this case, the error indication on line 7 is much more helpful than the information in the debug area.

Example 2: In Activity 02 where we changed the name of the IBAction method from actButton to actButton2, the application appeared to have crashed in the AppDelegate class. It is unlikely that we introduced any errors to that class since we did not modify any code in the AppDelegate. The error indication in the editor window tallies with the information in the debug area though. They both identified there was an unrecognized selector called actButton being referenced in ViewController.

The point is that you cannot simply look at where the application crashes. You have to exercise some judgement in how you interpret the various error messages. The first few times you encounter an error message, it may not appear to make sense. You may have to Google for answers, or try to deduce the meaning of the error. Once you have the solution for a particular error, you can apply the same or similar solution the next time you see that same error.

Breakpoints

https://help.apple.com/xcode/mac/current/#/dev9a374afc9

A breakpoint allows you to pause the execution of the computer program so that you can examine the state of the program, machine, and related data up to that point of execution. You can then proceed to step through the code to evaluate the effects of subsequent code execution.

You can set a breakpoint by clicking in the margin next to the line of code that you want to set a breakpoint. The breakpoint is indicated at the line number by a dark blue marker. Clicking on the breakpoint again disables the breakpoint, and the marker changes to a light blue color.

You can use the Breakpoint navigator to view, navigate, and manage all of the breakpoints in your project.

When you run an application, the application will halt at the location of the breakpoint that you set. The next line of code to be executed will be highlighted in green.

Stepping through code

Computer programs execute exceedingly fast, and that often makes it hard to catch an error. By stepping through the program line by line, the programmer is able to analyse the consequences of executing each line of code, which makes it easier to uncover the bug.

The debug bar in the debug area lets you control the execution of your running app. The debug area automatically opens when you build and run your app.

There are 4 buttons that control program execution. From the left, the buttons are:

  • Pause / continue execution - When the program is paused, pressing this button resumes execution of the program until the next breakpoint is reached. If there are no other breakpoints, the program runs as normal.
  • Step over - Execute the next statement, and pause. If the next statement calls a function, run the whole function and pause after returning from it. Will not step into the function the way step into does.
  • Step into - Execute the next statement, and pause. If the next statement calls a function, go into the function and pause before executing the first statement in the function.
  • Step out - Execute the rest of the current function and pause in the function that called this one.

Inspecting variable values

Variables are used extensively in any computer program, and for many bugs, the error comes as a result of an uninitialised variable, a variable holding an incorrect value, a variable being used incorrectly, using an incorrect variable, etc. Inspecting the value of a variable can help uncover a bug when you have a good idea what the value of the variable should be, as opposed to what the value of the variable is.

You can inspect a variable value in the variables view (the left panel) of the debug area of Xcode.
